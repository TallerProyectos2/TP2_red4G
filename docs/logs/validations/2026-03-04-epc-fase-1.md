# EPC Phase 1 Validation

- Date: `2026-03-04`
- Jira: `TP2-142`
- Machine: `tp2-EPC`

## Initial State

- `srsepc` was already running manually on the EPC with the command `sudo srsepc /home/tp2/.config/srsran/epc.conf`
- `epc.conf` still contained a relative HSS database path:
  - `db_file = user_db.csv`
- The live EPC log from `2026-03-03` showed the side effect of that relative path under a root-started process:
  - `Opened DB file: /root/.config/srsran/user_db.csv`

## Applied Change

- Backed up the live EPC config to `/srv/tp2/config/epc.conf.pre-fase1-20260304`
- Updated `/home/tp2/.config/srsran/epc.conf` to:
  - `db_file = /home/tp2/.config/srsran/user_db.csv`
  - `filename = /srv/tp2/logs/srsepc.log`
- Performed a controlled EPC-only restart:
  - stopped the existing `srsepc`
  - started `srsepc /home/tp2/.config/srsran/epc.conf`

## Runtime Evidence

- `grep -n '^db_file' /home/tp2/.config/srsran/epc.conf` now returns the absolute `user_db.csv` path
- `sed -n` confirmed:
  - `mme_bind_addr = 10.10.10.1`
  - `gtpu_bind_addr = 10.10.10.1`
  - `sgi_if_addr = 172.16.0.1`
- `sed -n` on `/home/tp2/.config/srsran/user_db.csv` confirmed at least two provisioned UEs: `ue1` and `ue2`
- `pgrep -af srsepc` after the restart returned a single live process:
  - `30003 srsepc /home/tp2/.config/srsran/epc.conf`
- `ss` confirms the EPC listeners are active:
  - `10.10.10.1:36412/SCTP`
  - `10.10.10.1:2152/UDP`
- `lsof -p 30003` shows `/srv/tp2/logs/srsepc.log` open by the running process
- Historic S1 evidence remains present in the EPC log:
  - `Received S1 Setup Request`
  - `Sending S1 Setup Response`
  - peer `srsenb01`
- A fresh 15-second observation window after the EPC restart produced no new S1 setup lines, so no new eNodeB connection attempt was observed during this validation window

## Result

The EPC side of Phase 1 is now consistent and restart-safe: the live LTE core process is running with the corrected absolute HSS database path, the required EPC-side addresses are bound, and the previously successful S1 handshake is preserved in the operational evidence. Fresh S1 re-establishment depends on the eNodeB initiating a new connection.
